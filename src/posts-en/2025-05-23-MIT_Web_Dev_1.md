How collaborative coding is now a critical issue that needs to be addressed

The most fundamental considerations include: independent copying of code, comparison of modified content, conflict resolution, and version control

These are exactly the problems that git aims to solve

Git records versions by tracking modifications

Some basic command-line operations

| Operation | Content               |
| ------- | ------------------ |
| `ls`    | List contents of a folder |
| `cd`    | Enter a specified folder     |
| `mkdir` | Create a folder         |

Initialization, creating a git repository:

~~~
git init
~~~

(Requires cd to the corresponding folder first)

When many files have been modified, and we want to package these changes and submit them to git's modification records, we can use `git add` to achieve this

![image-20250523161400265](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523161400265.png)

Using `git status` allows checking git's commit records

After using `git add`, the modified content is stored in the staging area cache, and only after `git commit` will it enter the repository's modifications

For example: the content after -m is some submission information

![image-20250523161719063](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523161719063.png)

![image-20250523161839841](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523161839841.png)

Notice the long string after the commit above is git's commit id, which will be used later

But there is still a problem: you have modified many contents, but the work requires you to first fix bugs in the original content. You need to switch back to the original work line from your current work line. How to solve this?

In git, it uses branch branching:

![image-20250523162251778](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523162251778.png)

We can use `git branch` to check the current branches:

![image-20250523162448137](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523162448137.png)

Currently only one master is seen

We can create a new branch using `git checkout`:

![image-20250523162605995](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523162605995.png)

![image-20250523163012392](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163012392.png)

After branching and modifying, how to return these changes to the main branch?

First check the branch `git branch`, then switch back to the main branch `git checkout main/master`, and finally merge: `git merge + branch name`

![image-20250523163713325](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163713325.png)

![image-20250523163828835](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163828835.png)

## 3 HTML Introduction
![image-20250523180610023](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523180610023.png)

HTML is a hypertext file, which acts as the skeleton of a webpage, while CSS serves as the flesh and clothing on top of the structure.

To grasp the main idea of HTML, it can be understood as a collection of adjacent boxes/containers.

For example, each red box in this image represents a container, and these containers collectively form the entire webpage:

![image-20250523180838239](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523180838239.png)

Next, let's examine a simple code example:

![image-20250523180918897](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523180918897.png)

Generally, HTML files typically include a `<!DOCTYPE html>` declaration to inform browsers that you are using the latest HTML version.

Continuing to examine the code above, we can easily identify some opening tags and closing tags, usually in the form of `<>` and `</>`, with the content between them forming the entire container.

For the HTML file above, it's evident that the entire HTML container contains two sub-containers: the `head` container and the `body` container.

Since HTML consists of containers, it is generally invalid to start a new container without closing the previous one:

![image-20250523181310914](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523181310914.png)

The `<html>` tag is the root container of the entire file, encapsulating all content within the HTML. The `<head>` container typically stores metadata, which often does not directly display on the webpage but serves auxiliary functions. The `<body>` contains all content that appears on the webpage.

Some commonly used general tags are as follows:

![image-20250523181537888](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523181537888.png)

Generally, these tags are not used directly but require assigning attributes:

~~~
<tagname attribute="value">
</tagname>
~~~

For example, for a link, the HTML is generally:

~~~
<a href="links here" str to print>
</a>
~~~

Regarding the `<div>` and `<span>` tags, their distinction lies in whether they cause line breaks:

![image-20250523182712417](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523182712417.png)

Why use `<div>`?

Because this helps us efficiently set the formatting for multiple tags at once.

For learning HTML, we don't need to memorize all tags. When needed, we can search for the appropriate tag using Google or the MDN Web Docs website.

Generally, it's not advisable to overuse `<div>`, and it's better to use tags suited for specific purposes.

> So: Don’t just use `<div>`. Use MDN to find the element you need!

Here, the div is the HTML container we select for decoration

The above CSS applies to HTML as shown below:

![image-20250523183752605](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523183752605.png)

![image-20250523183801210](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523183801210.png)

Sometimes we want to fine-tune the content inside a div, so we can add a class to the div:

~~~
<div class="info">Info</div>
~~~

Then, the CSS can be set as:

~~~
.info{
color:red;
font-family:Arial;
font-size:24pt;
}
~~~

This will only modify the corresponding div. Another approach is to use an id:

![image-20250523184109557](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523184109557.png)

What is the difference between these two?

1. An HTML div can have only one id, but can have multiple classes
2. An id corresponds to a unique container

![image-20250523184254929](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523184254929.png)

CSS has a hierarchical structure:

![image-20250523184344672](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523184344672.png)

The corresponding settings become more specific from bottom to top

Therefore, even if two settings differ, the more specific setting will be prioritized

After setting CSS, we need to import the CSS in HTML:

~~~
<head>
    <title>Title!</title>
    <link rel="stylesheet"  
          href="style.css" />
  </head>
~~~

For CSS blocks with specific functions, they are usually named with "u" as a prefix:

~~~css
.u-textCenter {
  text-align: center;
}
~~~
