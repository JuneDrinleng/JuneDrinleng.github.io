## 2 Git Basics

How to collaborate on code is now a problem that needs to be solved closely

The most basic points to consider are: independent copies of code, comparison of modified content and conflict resolution, and version control.

These are exactly the problems that git is trying to solve

git implements recorded versions by tracking changes

Some basic command line operations

| Operation | Content |
| ------- | ------------------ |
| `ls` | List the contents of a folder |
| `cd` | Enter the specified folder |
| `mkdir` | Create folder |

Initialize and create a git repository:

~~~
git init
~~~

(You need to cd to the corresponding folder first)

When many files of the code have changed, when we want to package these changes and submit them to the git change record, we can use `git add` to achieve this

![image-20250523161400265](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523161400265.png)

Use `git status` to check the git commit record

After using `git add`, the modified content is stored in the staging area cache. Only after we `git commit` will we enter the modification of the warehouse.

For example: Some information submitted by the content after -m here

![image-20250523161719063](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523161719063.png)

Now look at `git log`:

![image-20250523161839841](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523161839841.png)

Notice that the long list of things after the commit above is the commit id of git. We will use this commit id when we want to use it later.

But we still face a problem, that is, you have modified a lot of content, but the work requires you to fix the bugs in the original content first. You need to return from your current work line to the original work line. How should you solve this?

In git, the branch method is used:

![image-20250523162251778](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523162251778.png)

Each node is an operation, so you can see that the above three operations are branched from the last home page. Different people make different modifications, and git will also record these branches.

We can use `git branch` to see what branches are available now:

![image-20250523162448137](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523162448137.png)

See that there is currently only one master

Then you can create a new branch through `git checkout`:

![image-20250523162605995](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523162605995.png)At this point, a test branch is created and switched from master to test.

Next, you can make changes on this branch

![image-20250523163012392](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163012392.png)

After the branch is generated and modified, how should these modifications be put back to the main branch?

First check the branch `git branch`, then switch back to the main branch `git checkout main/master` and finally merge: `git merge + branch name`

The final warehouse structure becomes:

![image-20250523163511844](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163511844.png)

But the above content is only implemented locally. If multiple people need to be synchronized in different places, a remote service is needed, usually using GitHub. How to implement this:

First of all, the code exists in a remote server. The first thing everyone needs to get is a copy of the entire warehouse:

![image-20250523163713325](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163713325.png)

This usually uses `git clone`

Then modify and commit locally:

![image-20250523163750840](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163750840.png)

Finally, you need to use `git push` to submit the contents of the commit in local git to the remote one:

![image-20250523163828835](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163828835.png)

Others who want to compare their own version with the version on the remote server should use `git fetch`:

![image-20250523163909460](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523163909460.png)

To update other people's changes from the remote server to your local git, use `git pull`

`git reset --hard` will clear all uncommitted temporary storage (directly reset all add content)

## 3 Introduction to HTML

![image-20250523180610023](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523180610023.png)

HTML is a hypertext document, it is like the skeleton of a web page, while CSS is the flesh and clothing on the stock price.

For the main idea of understanding HTML, it can be understood as a bunch of adjacent boxes/containers

For example, each red frame in this picture is a container, and these containers are combined to form the entire web page:

![image-20250523180838239](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523180838239.png)

Next let's look at a simple example of code:

![image-20250523180918897](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523180918897.png)

Generally, html files usually contain a `<!DOCTYPE html>` to tell the browser that you are using the latest html.

Then continue to look at the above code and we can easily find some opening tags and some closing tags, usually in the form of <> and </>, then the entire container is between them.

So for the above html file, it is easy to find that there are two sub-containers in the entire html container, namely the head container and the body container.

Since HTML is a bunch of containers, generally speaking, it is not allowed to start a new container before the following containers are finished:

![image-20250523181310914](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523181310914.png)

`<html>` is the root container of the entire file, a container that wraps all the content of the entire html. The `<head>` container generally stores some metadata. They are often not displayed directly on the web page, but will have some other auxiliary functions. `<body>` contains all content that will appear in the web page.

Some commonly used generic tags are as follows:

![image-20250523181537888](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523181537888.png)

Generally these tags will not be used directly but there will be some attributes that need to be assigned:

~~~
<tagname attribute="value">
</tagname>
~~~

For example, for a link, its html is generally:

~~~
<a herf="links here" str to print>
</a>
~~~

## 4 Introduction to CSS

CSS = a list of descriptions

The following is the basic unit of a css![image-20250523183514187](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523183514187.png)

The div here is the html container we choose to decorate.

The above css acts on html as follows:

![image-20250523183752605](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523183752605.png)

![image-20250523183801210](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523183801210.png)

Sometimes we want to more finely control the content inside the div, then we can add a class to the div:

~~~
<div class="info">Info</div>
~~~

Then CSS only needs to be set:

~~~
.info{
color:red;
font-family:Arial;
font-size:24pt;
}
~~~

You can only modify the corresponding div. Another way is to use the id:

![image-20250523184109557](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523184109557.png)

So what's the difference between the two?

1. An html div can only have one id, but it can have many categories.
2. An id can only correspond to one and only one container.

![image-20250523184254929](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523184254929.png)

There are certain levels for CSS:

![image-20250523184344672](https://raw.githubusercontent.com/JuneDrinleng/JuneDrinleng.github.io/main/assets/img/image-20250523184344672.png)

The corresponding ones become more and more specific from bottom to top.

So even if the two settings are different, the setting will be based on the higher specific setting result first.

Then after we set the css, we need to introduce Chinese css into html:

~~~
<head>
    <title>Title!</title>
    <link rel="stylesheet"
          href="style.css" />
  </head>
~~~

For some specific blocks, the css is generally named starting with u:

~~~css
.u-textCenter {
  text-align: center;
}

~~~
